Why may your team fail with AngularJS?

1. Not a usual web project

When your team (or you) decide to use AngularJS as a framework for your project you make a step into Single-Page Application world where usual skills for front-end developers are very different by value.

In classic web projects you can have guys who will be able to write data access code with database communication but also them should work with client-side techonologies like Javascript, CSS and HTML. Usually them are dislike it. Correct solution here is introduce specialization between team members. One group is working on data access and business logic, but second group is working on representation layer.

In classic projects (non SPA) it works not good (even with strong MVC separation in architecture) because anyway such groups share same code. And no way to split it because them share one server-side language (like PHP, Ruby or C#) for them tasks. Tasks are different, but language is one. Javascript plays second role because templates are produced by server. It generates a lot load to the server but it is an another story.

Team structure difference:

Classic:

A team for PHP/Ruby SQL/NoSQL Javascript HTML/CSS UI/Design

SPA:

A server-side team for PHP/Ruby SQL/NoSQL

A client-side team for Javascript HTML/CSS UI/Design

To make long story short here is AngularJS for. It can replace all server-side template systems and move this tasks to the client browser. Server will be used only for data-access. Your teams will get full separation of development and will use right tools for each task. Client-side team will work with Javascript (or any dialect like Coffeescript) as a main language, but your server-side team will use its favorite server-side language like PHP/Ruby/etc.

2. Client-side is important

Many time ago teams thought server-side is more important than client-side because it should save data and data is all. But for current time we live in a startup-world where customer discover is the most important to achieve project goals. And here client-side is a place where your clients meets your goods and services.

So, how better your client-side will be so bigger amount of clients you will get. And here it is important to handle all complex cases of logic on client-side, but usual jquery-coders can't do it in right way.

Because them will be able to know full spectrum of new techonologies which are difficult to learn at once. For example, them will need to know: Promises, ViewModel, Model-View-ViewModel, SOLID principles, REST design, Test-Driven Development, Domain-Driven Development, Functional code style and many other stuff...

Just because client-side is your face it is important to have strong coders on board to produce stable releases or anybody who will do a mentoring job.

Usual tasks for client-side:

  • Prepare UI concept
  • Define transitions
  • Draw design
  • Split design into small pieces of html/css
  • Introduce view models based on user expected behaviour
  • Prepare unit-tests and mocks
  • Write client-side business logic to connect views and models
  • Write data access logic to retrieve data from server-side
  • Write Selenium tests to verify requirements
  • Turn on build process
  • Turn on deployment process
